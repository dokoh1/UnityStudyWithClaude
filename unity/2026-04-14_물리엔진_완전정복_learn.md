## Rigidbody와 Collider의 관계

### 각각의 역할

- **Rigidbody**: "이 오브젝트는 물리 법칙의 영향을 받는다"
  - 속도(velocity), 각속도(angular velocity), 질량(mass) 등
  - 중력, 충격력, 드래그가 적용됨

- **Collider**: "이 오브젝트의 물리적 형태"
  - 충돌 감지를 위한 도형
  - 시각적 메시와는 별개 (충돌용으로 단순화)

### 네 가지 조합

```
① Collider만 → Static Collider (움직이지 않는 벽, 바닥)
② Rigidbody + Collider → Dynamic Rigidbody (중력/충격 영향)
③ Rigidbody(Kinematic) + Collider → 물리 영향 없지만 충돌 가능
④ 둘 다 없음 → 물리 시스템 완전 무시 (단순 장식용)
```

### 각속도와 드래그

- **velocity**: 선속도, 직선 이동 속도 (m/s)
- **angularVelocity**: 각속도, 회전 속도 (rad/s, 2π = 1바퀴)
- **drag**: 공기 저항 같은 감쇠 (0 = 영원히, 100 = 즉시 멈춤)
- **angularDrag**: 회전 감쇠

Drag 실무 가이드:
- 일반 박스: drag 0, angularDrag 0.05
- 공중 공: drag 0.3~0.5 (공기 저항)
- 물속: drag 2~5
- 캐릭터: drag 0 (직접 제어)

### 왜 Static Collider를 움직이면 안 되는가?

PhysX가 공간 분할 자료구조(AABB Tree)로 관리:
- Static은 "안 움직인다" 전제로 트리 구성
- 움직이면 매번 재구성 → 심각한 성능 문제
- 움직여야 하면 Rigidbody(Kinematic) 추가

## Unity 물리의 내부 구현

### 실제 구조

Unity는 직접 물리 엔진을 구현한 게 아니라 **NVIDIA PhysX를 래핑**:

```
게임 코드 (C#)
    ↓
Unity C++ Engine (Binding)
    ↓
NVIDIA PhysX (실제 계산)
  ├── PxRigidDynamic (Rigidbody)
  ├── PxShape (Collider)
  ├── PxScene (물리 세계)
  ├── PxBroadphase (공간 분할)
  └── PxNarrowphase (정확한 충돌)
```

### 공간 분할 자료구조

**문제**: 1000개 오브젝트 → 1000×999/2 = 499,500번 체크 필요!

**해결**: 멀리 떨어진 애들은 체크 안 하도록 공간 분할

**대표 구조**:
- Uniform Grid: 씬을 격자로 나눔
- Quadtree/Octree: 재귀적으로 4등분/8등분
- **AABB Tree** (PhysX 사용): 각 오브젝트를 AABB 박스로 감싸고 계층적 트리

**AABB**: Axis-Aligned Bounding Box = 축에 정렬된 경계 상자. 실제 메시 주위에 간단한 박스를 씌워서 빠른 1차 필터링.

### Broadphase와 Narrowphase

충돌 검사는 2단계:
1. **Broadphase (광역)**: AABB Tree로 가까운 쌍 찾기, O(n log n)
2. **Narrowphase (상세)**: 실제 Collider 모양으로 정확한 판정

### Static/Dynamic 분리 관리

PhysX는 내부적으로 두 개의 트리:
- **Static Tree**: 업데이트 거의 없음, 빌드 시 한 번만
- **Dynamic Tree**: Dynamic + Kinematic, 매 프레임 효율적 업데이트

Static을 움직이면 Static Tree 재구성 → 폭탄!
Kinematic은 Dynamic Tree에서 관리 → 효율적!

## FixedUpdate에서 물리를 처리하는 이유

### Update vs FixedUpdate

- **Update**: 매 프레임 호출, 간격 불규칙 (60fps = 16ms, 30fps = 33ms)
- **FixedUpdate**: 고정 간격 (기본 0.02초 = 50Hz), PC 성능 무관

### 왜 고정 간격이 필요한가

**이유 1: 수치 시뮬레이션의 정확성**
- 물리 공식: `새 위치 = 현재 위치 + 속도 × 시간간격`
- 프레임 간격이 다르면 누적 오차 발생
- 30fps와 60fps에서 결과가 달라짐

**이유 2: 터널링 방지**
- 프레임 간격이 크면 빠른 물체가 벽을 "건너뜀"
- FixedUpdate가 빠를수록 촘촘히 체크

**이유 3: 재현성**
- 리플레이, 네트워크 동기화
- 같은 입력 → 같은 결과 필요
- 프레임레이트 독립적이어야 함

## Kinematic이 뭔가

### 정확한 정의

**Kinematic = "개발자가 움직임을 완전히 제어"**

- "Unity야, 물리 계산하지 마. 내가 알아서 움직일게"
- 물리 영향 안 받음 (중력 X, 충격 X)
- 하지만 충돌 감지는 됨
- transform 또는 MovePosition으로 이동

### 특별한 동작

1. 다른 Dynamic Rigidbody를 밀어낼 수 있음 (영향 줌)
2. 다른 물체의 힘을 받지 않음 (영향 안 받음)
3. 충돌 이벤트는 정상 발생
4. 자동 반응 없음 (벽 뚫고 지나감, 직접 구현 필요)

### Kinematic끼리 충돌 안 되는 이유

- 둘 다 "내가 주도권" 주장
- 어느 쪽이 어떻게 반응할지 애매
- 기본은 충돌 무시, 필요 시 설정으로 활성화

### 실무 용도

- 움직이는 플랫폼, 문, 엘리베이터
- AI 제어 캐릭터
- 애니메이션 기반 이동
- 네트워크 동기화 오브젝트

## 이동 방식 5가지 총정리

### 1. Transform 직접 이동
```csharp
transform.position += moveDir * speed * Time.deltaTime;
```
- 물리 완전 무시, 가장 빠름, 완전 제어
- 충돌 없이 벽 뚫음
- 적합: UI, 컷씬, 장식용, 프로토타입

### 2. Rigidbody.velocity / AddForce
```csharp
rb.AddForce(moveDir * force);
```
- 완전한 물리 시뮬레이션
- 자동 충돌 처리, 관성, 마찰
- 제어 어려움
- 적합: 굴러가는 공, 차량, 물리 퍼즐, 발사체

### 3. Rigidbody.MovePosition (Kinematic 이동)
```csharp
rb.MovePosition(rb.position + moveDir * speed * Time.fixedDeltaTime);
```
- Kinematic의 이동 방식
- Interpolation 자동 (부드러움)
- 충돌 이벤트 정상, 자동 반응 없음
- 적합: 움직이는 플랫폼, 문, AI 캐릭터

### 4. CharacterController.Move
```csharp
cc.Move(delta);
```
- 캐릭터 특화, 물리 없이 충돌 처리
- 경사/계단 자동 (slopeLimit, stepOffset)
- 캡슐 모양 고정
- 적합: 3인칭 액션, FPS

### 5. Kinematic + Raycast (고급)
- 완전 커스텀 제어
- 직접 충돌 체크
- 적합: Celeste, Hollow Knight 같은 정밀 플랫포머

## CharacterController

### 정의
Rigidbody의 대안. 물리 시뮬레이션 없이 캐릭터 이동에 특화.

### 특징
- 내부적으로 CapsuleCollider + 특화 로직
- `isGrounded` 프로퍼티 내장
- slopeLimit, stepOffset로 경사/계단 자동 처리
- 물리 오브젝트를 밀 수 없음

### 예시 코드
```csharp
void Update()
{
    if (cc.isGrounded && velocity.y < 0) velocity.y = -2f;

    Vector3 input = new Vector3(Input.GetAxis("Horizontal"), 0, Input.GetAxis("Vertical"));
    if (cc.isGrounded && Input.GetKeyDown(KeyCode.Space))
        velocity.y = jumpSpeed;

    velocity.y += gravity * Time.deltaTime;
    cc.Move((input * moveSpeed + velocity) * Time.deltaTime);
}
```

## 9. 왜 입력은 Update에서 감지?

### 문제 상황

유저가 Space를 짧게 (10ms) 눌렀다 떼면:
- **Update (16ms 간격)**: 눌린 프레임에서 GetKeyDown 감지 ✅
- **FixedUpdate (20ms 간격)**: 이미 떼진 시점에 체크 → 놓침 ❌

### 이유

Unity의 입력 버퍼는 "프레임 단위"로 업데이트됨:
- `GetKeyDown`은 Update 프레임 기준으로 정확
- FixedUpdate에서는 타이밍에 따라 놓칠 수 있음

### 올바른 패턴 (플래그 방식)

```csharp
private bool jumpPressed = false;

void Update()  // 입력 감지
{
    if (Input.GetKeyDown(KeyCode.Space))
        jumpPressed = true;
}

void FixedUpdate()  // 물리 적용
{
    if (jumpPressed)
    {
        rb.AddForce(Vector3.up * jumpForce, ForceMode.Impulse);
        jumpPressed = false;
    }
}
```

### 고성능 PC에서도 문제없는 이유

- Update가 FixedUpdate보다 5배 많이 호출돼도
- 깃발은 한 번만 올라가고 한 번만 내려감
- 점프는 다음 FixedUpdate에 정확히 1번만 적용

## 10. Collision Detection Mode의 내부 구현

### Discrete (기본값)
**방식**: 각 FixedUpdate마다 현재 위치만 체크
- 구현: AABB 박스 교차 판정 (6번 비교)
- 속도: 가장 빠름
- 단점: 빠른 물체가 얇은 벽 뚫음 (터널링)
- 적합: 일반 오브젝트

### Continuous
**방식**: 이전 위치 → 새 위치의 "경로 전체" 체크
- 구현: Swept Volume 계산 (이전 AABB + 새 AABB를 연결)
- TOI (Time of Impact) 계산
- 비용: Discrete 대비 ~2배
- 제약: Static/Mesh Collider에만 sweep
- 적합: 빠른 발사체

### Continuous Dynamic
**방식**: Continuous + 다른 Dynamic Rigidbody도 sweep
- 자기 경로와 다른 Dynamic 경로 교차 계산
- 비용: ~3~4배
- 가장 정확
- 적합: 빠른 차량, 중요한 충돌

### Continuous Speculative (2019+)
**방식**: 이동 예상 영역을 AABB로 확장, Solver가 반응
- 경로 추적 대신 예측
- 회전에도 대응
- Continuous보다 저렴 (~1.2배)
- 가끔 false positive
- 적합: 회전하는 빠른 물체, 캐릭터

### 성능 차이 (1000개 기준)
- Discrete: 1x
- Speculative: 1.2x
- Continuous: 1.5~2x
- Continuous Dynamic: 3~4x

## 11. 왜 OnCollision/OnTrigger는 Rigidbody 필요?

### 핵심: 누가 "검색의 주체"인가

PhysX의 동작:
- Rigidbody를 가진 오브젝트가 "주변에 뭐 있나" 검색
- Static Collider는 "찾아지는 대상"일 뿐, 스스로 검색 안 함

### 비유
- Rigidbody = 레이더 장착 자동차 (주변 스캔)
- Static Collider = 도로 위 장애물 (레이더 없음)
- 둘 다 레이더 없으면 서로 감지 불가

### 충돌 매트릭스

|  | Static | Kinematic | Dynamic |
|---|---|---|---|
| Static | ❌ | ✅ | ✅ |
| Kinematic | ✅ | ❌(*) | ✅ |
| Dynamic | ✅ | ✅ | ✅ |

(*) Kinematic끼리는 설정으로 활성화 가능

### 규칙
**둘 중 최소 하나는 Rigidbody (Kinematic 포함)**

### 왜 이런 설계인가
성능 최적화:
- Static 500개끼리 체크하면 124,750번 낭비
- Static은 안 움직이니 결과가 항상 같음
- "움직이는 것(Rigidbody)만 검색" → 극적 성능 향상

### 해결: Rigidbody 없이 이벤트 받고 싶으면
Kinematic Rigidbody 추가 → "움직이지 않지만 감지 능력 있음"

## Trigger vs Collision

### Is Trigger 체크박스
- **체크 안 됨**: Solid Collider, 물리적 반응 발생, OnCollision 이벤트
- **체크 됨**: Trigger 영역, 통과함, OnTrigger 이벤트

### 사용 구분
- **Collision**: 벽, 바닥, 박스, 튕기는 공
- **Trigger**: 체크포인트, 아이템 영역, 감지 범위, 데미지 존, 포털

### 이벤트 발생 조건
- 두 오브젝트 모두 Collider
- 둘 중 하나는 Rigidbody (Kinematic 포함)
- Trigger인지 여부에 따라 OnCollision/OnTrigger 선택

## Unity 물리 엔진이 저평가받는 이유

### 주요 비판점

1. **일관성 없는 동작**: 프레임레이트 의존, 리플레이 100% 재현 안 됨
2. **캐릭터 컨트롤러 미흡**: CharacterController와 Rigidbody 둘 다 한계
3. **멀티플레이어 동기화 어려움**: 결정론적(deterministic)이지 않음
4. **2D 물리 제약**: Box2D 통합이 매끄럽지 않음
5. **대량 물리 성능**: 1000+ Rigidbody에서 급락
6. **고급 기능 부족**: Soft Body, Fluid, 파괴 시뮬레이션 없음

### 직접 구현이 필요한 경우

- 격투 게임 (결정론적 물리 필요)
- 경쟁 멀티플레이어 (정확한 동기화)
- 복잡한 캐릭터 컨트롤러
- 특수 메카닉 (가변 중력, 벽타기)

### 직접 구현 방법
1. CharacterController + 수동 이동 계산
2. Kinematic Rigidbody + Raycast
3. 수학적 시뮬레이션 직접 구현
4. 서드파티 물리 엔진 (Havok, Unity.Physics)
5. DOTS + Unity.Physics (결정론적)

## 오답 노트

1. **Rigidbody와 Collider는 역할이 다름** — Rigidbody는 "물리 영향 받음", Collider는 "물리적 형태". 둘의 조합으로 4가지 상태.

2. **Static Collider를 절대 움직이면 안 됨** — PhysX의 Static Tree가 매번 재구성되어 성능 폭탄. Kinematic을 쓸 것.

3. **Kinematic의 정확한 의미** — 단순 "움직이는 Collider"가 아니라 "개발자가 움직임을 완전 제어, 물리 영향 받지 않음". Kinematic끼리 충돌 안 되는 것도 이 때문.

4. **FixedUpdate는 실제 시간 기반** — 프레임레이트 독립적. 물리 계산은 정확한 시간 간격이 필수.

5. **입력은 Update, 물리는 FixedUpdate** — GetKeyDown을 FixedUpdate에서 쓰면 놓칠 수 있음. 플래그로 중개.

6. **Continuous는 Swept Volume 계산** — 단순 박스 교차가 아니라 경로 전체를 추적하는 복잡한 계산이라 느림.

7. **Rigidbody = "검색의 주체"** — Static은 찾아지기만 함. 이벤트를 받으려면 한쪽은 Rigidbody 필요. Kinematic도 OK.

8. **Unity 물리가 저평가받는 이유는 구현이 아니라 통합** — PhysX 자체는 AAA급. Unity의 결정론적이지 않은 처리, 캐릭터 컨트롤 한계가 문제.
