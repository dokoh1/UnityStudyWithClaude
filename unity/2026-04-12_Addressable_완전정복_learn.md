## 이야기의 시작: AssetBundle이 뭐야?

> "AssetBundle에 대해서 잘 모르겠어. Addressable은 주소로 참조한다는데, 그러면 빌드할 때 실제 데이터는 어디로 가는 거야?"

이 질문이 전체 대화의 출발점이었다. 답은 단순했다 — **빌드 과정에서 에셋이 별도의 `.bundle` 파일로 따로 분리되어 저장된다.**

### 일반 빌드 vs Addressable 빌드

일반 빌드는 에셋이 `sharedassets.assets`에 전부 묶여서 씬 로드 시 전부 메모리에 올라간다. Addressable 빌드는 에셋이 `StreamingAssets/aa/` 폴더에 `.bundle` 파일들로 따로 분리되고, 코드에는 "주소(문자열)"만 있어서 필요할 때만 로드한다.

```
빌드 파일 구조:

  Build/
  ├── game.exe
  ├── game_Data/
  │   └── sharedassets0.assets   ← 일반 에셋은 여기
  └── StreamingAssets/aa/         ← ⭐ Addressable 에셋은 여기!
      ├── catalog.json            ← 주소 → 번들 매핑
      └── *.bundle                ← 실제 에셋 파일들
```

---

## AssetBundle과 Addressable의 관계

**Addressable은 AssetBundle 위에 만들어진 상위 시스템**이다.

- AssetBundle = 에셋을 묶은 파일 (.bundle) — 택배 상자
- Addressable = AssetBundle을 쉽게 쓸 수 있게 해주는 래퍼 — 택배 서비스

AssetBundle을 직접 쓰면 번들 빌드, 경로 관리, 의존성 추적, 해제 타이밍, 서버 다운로드, 캐싱을 전부 수동으로 해야 한다. Addressable은 이 모든 것을 자동화한다. 신규 프로젝트라면 무조건 Addressable.

## 첫 번째 의문: "빌드 파일 용량은 똑같은 거 아니야?"

### 전부 로컬 그룹이면 맞다

로컬 그룹만 쓰면 빌드 크기는 같다 (500MB → 500MB). 하지만 **메모리 사용량은 줄어든다**. 일반 빌드는 씬 로드 시 참조된 에셋이 전부 메모리에(500MB), Addressable은 필요한 번들만 메모리에(50MB).

### 원격 그룹을 쓰면 빌드 크기도 줄어든다

로컬 + 원격 분리 시: 빌드 50MB + CDN 450MB = 총 500MB. 앱스토어 설치 크기는 50MB로 축소. 나머지는 게임 시작 후 다운로드.

```
정리:
  전부 로컬:  빌드 같음, 메모리 ↓
  로컬+원격:  빌드 ↓, 메모리 ↓
```

## 두 번째 의문: "레벨마다 다운로드 창 뜨는 게임 못 봤는데?"

### 실제 게임에서 쓰는 다운로드 패턴

**패턴 1: 첫 실행 시 대규모 다운로드 (가장 흔함)**

앱스토어에서 50MB 설치 → 게임 시작 → "데이터를 다운로드합니다" 화면에서 4GB 다운로드 → 이후 플레이. 원신, 블루아카이브, 명일방주, 쿠키런: 킹덤 등 대부분의 모바일 게임이 이 패턴. 앱스토어 크기 제한과 설치율 때문.

**패턴 2: 백그라운드 미리 다운로드**

레벨 3 플레이 중에 레벨 4, 5 에셋을 조용히 미리 다운로드. 유저가 레벨 4에 도달하면 이미 받아져 있어서 로딩이 짧음. "로딩 화면에서 실제로는 다운로드/로드하는 중"인 경우도 많다.

**패턴 3: 이벤트/업데이트 시 알림 다운로드**

"할로윈 이벤트 업데이트 45MB" 팝업. 앱 업데이트 없이 서버에서만 다운로드. 메이플스토리M, 쿠키런 같은 게임에서 자주 보임.

**패턴 4: 스트리밍 (오픈월드)**

플레이어 주변 지역만 로드, 멀어지면 해제. 원신, 젤다 같은 게임. Addressable만으로는 부족하고 커스텀 스트리밍 시스템 필요.

## 세 번째 의문: "로드가 완료 안 된 상태에서 에셋을 쓰면?"

3가지 패턴이 있다:

1. **로딩 완료까지 대기** — 스피너 표시 후 한 번에 표시. 안전하지만 0.5초 대기.
2. **플레이스홀더 → 교체** — 즉시 기본 아이콘 표시, 로드되면 교체. 가장 흔함. 페이드인으로 부드럽게.
3. **프리로드** — 게임 시작 시 미리 로드해서 Dictionary에 캐시. 최고의 반응성, 메모리 미리 사용.

실무에서는 자주 쓰는 UI(인벤토리)는 프리로드 + 플레이스홀더(보험), 가끔 쓰는 UI(도감)는 플레이스홀더만.

## Addressable을 언제 쓰는가 — 케이스별

### 안 써도 되는 경우
- 소규모 게임 / 프로토타입 (에셋 50MB 이하)
- 게임잼 / 학생 프로젝트
- 에셋이 적은 2D 게임

### 써야 하는 경우
- **모바일 게임** — 메모리 제한 + 앱스토어 크기 제한
- **라이브 서비스** — 앱 업데이트 없이 콘텐츠 추가
- **대규모 에셋 게임** — 1GB 이상
- **DLC / 유료 콘텐츠** — 구매한 유저만 다운로드
- **멀티플랫폼** — 플랫폼별 다른 에셋 버전

## 네 번째 의문: "원격 Addressable은 서버를 새로 만드는 거야?"

### 게임 서버가 아니라 파일 저장소

```
서버를 만드는 게 아니라 "파일 저장소(CDN)"에 파일을 올리는 것!

사용 가능한 CDN 서비스:
├── AWS CloudFront
├── Google Cloud CDN
├── Cloudflare R2
├── Unity Cloud Content Delivery (Unity 공식)
└── 심지어 GitHub Pages, 구글 드라이브 (테스트용)
```

"서버를 만든다" = 게임 로직 처리 서버 (멀티플레이어). 복잡하고 비쌈.
"원격 Addressable 서버" = 파일 저장소. CDN에 가입 → 파일 업로드 → URL만 Addressable 설정에 입력. 서버 코드 작성 불필요.

### "왜 구글은 플레이스토어에서 안 받고 CDN을 따로 쓰는가?"

플레이스토어의 제약:
- 기본 150MB 제한
- 심사 1~3일
- 유저가 직접 업데이트 버튼 눌러야 함
- 긴급 패치 불가

CDN의 장점:
- 크기 제한 없음 (원신은 수십 GB)
- 즉시 배포
- 강제 업데이트 가능
- 지역/유저별 제어 가능

## 다섯 번째 의문: "디스크와 메모리는 다른 개념인가?"

**완전히 다른 개념이다.**

```
디스크 (Storage):
├── SSD, HDD, eMMC, UFS
├── 전원 꺼져도 유지 (비휘발성)
├── 용량 큼 (수십 GB ~ 수 TB)
├── 속도 느림 (100MB/s ~ 3GB/s)
└── 파일 단위로 저장

메모리 (RAM):
├── DDR4, DDR5
├── 전원 꺼지면 사라짐 (휘발성)
├── 용량 작음 (수 GB ~ 수십 GB)
├── 속도 매우 빠름 (~20GB/s)
└── 실행 중인 프로그램과 데이터
```

**비유**: 디스크 = 책장 (영구 보관), 메모리 = 책상 (지금 읽는 책). 책장에서 책상으로 가져오는 게 "로드". 책상이 꽉 차면 돌려놔야 함 (해제).

**왜 둘 다 필요한가**: 메모리는 휘발성이고 비싸고 작음. 디스크는 너무 느려서 CPU가 직접 실행 불가. 그래서 파일은 디스크에 저장, 사용할 때만 메모리로 복사.

### Addressable의 디스크-메모리 흐름

```
① CDN → 디스크 (다운로드): 초~분 단위, 최초 1회
② 디스크 → 메모리 (로드): 밀리초 단위, 필요할 때마다
③ 메모리 해제: 즉시, 디스크 파일은 그대로
```

## 여섯 번째 의문: "기기 업데이트 vs 플레이스토어 업데이트의 차이는?"

### 코드와 데이터의 경계

```
기기에서만 업데이트 가능 (CDN):
├── 에셋 (이미지, 사운드, 모델)
├── ScriptableObject 데이터 (밸런스)
├── 레벨 데이터
├── 이벤트 콘텐츠
└── 기존 시스템을 활용하는 새 콘텐츠

플레이스토어 업데이트 필수:
├── 게임 코드 (C# 로직 추가/수정)
├── Unity 엔진 버전 업
├── 새로운 플러그인
├── 권한 변경
└── 완전히 새로운 시스템 (PVP, 길드 등)
```

### 예시로 구분

- "검의 공격력 100 → 150" → 데이터 → CDN ✅
- "크리티컬 시스템 추가" → 코드 → 플레이스토어 ❌
- "새 스킨" → 모델/텍스처만 → CDN ✅
- "채팅 시스템 추가" → 코드 → 플레이스토어 ❌

### 잘 설계된 라이브 서비스

범용 이벤트 시스템을 코드로 미리 만들어두고, 이벤트의 "내용"만 데이터로 받아옴. 1년에 1~2번만 플레이스토어 업데이트, 나머지는 CDN으로 해결.

## 일곱 번째 의문: "로컬과 원격은 어떻게 나뉘는가?"

**개발자가 Addressable 에디터에서 그룹별로 설정하는 것.**

```
Window > Asset Management > Addressables > Groups

▼ Default_Local         Build Path: Local   ← 앱에 포함
▼ Events_Remote         Build Path: Remote  ← CDN에 업로드
▼ Level3_Remote         Build Path: Remote
```

### 분류 기준

**로컬 그룹**:
- 반드시 필요한 것 (로딩, 기본 UI)
- 오프라인에서도 필요
- 크기 작고 자주 쓰는 것
- 튜토리얼, 스타터 콘텐츠

**원격 그룹**:
- 크기 큰 것 (캐릭터 모델)
- 일부 유저만 쓰는 것 (특정 캐릭터)
- 자주 업데이트 (이벤트, 밸런스)
- DLC, 유료 콘텐츠
- 지역/언어별 콘텐츠

## 여덟 번째 의문: "디스크에서 메모리로 로드하는 시간은?"

```
에셋 크기별 로드 시간 (모바일 기준):

텍스처 1MB:   ~5~20ms
텍스처 10MB:  ~50~150ms
텍스처 50MB:  ~300~800ms

유저 체감:
├── 50ms 이하: 즉시 (플레이스홀더 없이도 OK)
├── 50~500ms: 플레이스홀더 → 교체 패턴
└── 500ms 이상: 명시적 로딩 화면 필요
```

영향 요소: 디스크 속도(SSD vs HDD), 에셋 크기, 압축 방식(LZ4 vs LZMA), 동기/비동기.

**첫 로드(CDN 다운)은 초~분 단위**, 캐시된 로드(디스크에서)는 ms 단위. 그래서 미리 다운로드가 중요.

## 핵심 결론: 두 가지 이해

### 1. 플레이스토어 앱의 구성

"초기 UI, 로딩 화면, 로그인 화면, 설치를 위한 코드"가 정확한 이해다.

플레이스토어 설치 파일에 들어가는 것:
- 게임 코드 (C# 로직, Unity 엔진 런타임)
- 초기 화면 에셋 (스플래시, 로딩, 로그인)
- 기본 에러 처리, 설정, 공지사항
- Addressable 카탈로그 (주소 목록)
- 로컬 번들 (선택, 최소한의 핵심 에셋)

게임 콘텐츠는 거의 없음. "게임을 시작할 수 있는 최소한"만. 나머지는 게임 실행 후 CDN에서 다운로드. 비유하면 "차체와 엔진만 있는 택시".

### 2. Addressable의 본질

**"디스크에 있는 값들을 주소로 저장해서 필요할 때마다 메모리에 올리는 것"** — 정확히 맞다.

```
3단계 역할:
① CDN → 디스크 (다운로드, 최초 1회)
② 디스크 → 메모리 (로드, 필요할 때)
③ 메모리 해제 (다 썼을 때, 디스크는 유지)

코드의 "주소"가 하는 일:
1. catalog.json 조회 → 번들 위치 파악
2. 번들을 메모리에 로드
3. 번들 안에서 에셋 꺼내서 객체 생성
4. 반환
```

**일반 참조 = 자동, 편하지만 제어 불가**
**Addressable = 수동, 귀찮지만 완전 제어**

대규모 프로젝트에서는 수동 관리의 이점이 압도적.

## 오답 노트

이 세션에서 새로 정리된 인사이트:

1. **"빌드 크기 = 메모리 사용량"이 아니다** — 로컬 Addressable은 빌드 크기는 같지만 메모리는 줄어든다. 이 둘은 다른 지표.

2. **Addressable 서버 ≠ 게임 서버** — CDN은 파일 저장소일 뿐, 게임 로직을 처리하는 서버가 아니다. CDN 서비스에 가입해서 파일 업로드만 하면 된다.

3. **실제 게임의 다운로드 패턴은 "첫 실행 대규모 다운로드"** — 레벨마다 다운로드 창이 뜨는 게 아니라, 앱 설치 직후 한 번에 받고 이후엔 투명하게 처리. 원신, 블루아카이브 등이 대표적.

4. **디스크 vs 메모리 = 비휘발성 vs 휘발성** — 디스크는 파일을 영구 저장, 메모리는 실행 중인 것을 임시 저장. Addressable은 CDN→디스크→메모리의 3단계 데이터 흐름을 관리.

5. **코드는 플레이스토어, 데이터는 CDN** — 게임 로직 수정은 앱 업데이트 필수, 에셋/설정 변경은 CDN만으로 가능. 잘 설계된 게임은 이 경계를 활용해 라이브 운영.

6. **플레이스토어 앱의 본질** — "게임 전체"가 아니라 "게임을 시작할 수 있는 최소한의 뼈대". 실제 콘텐츠는 CDN에서 다운로드. 택시의 차체와 엔진만 앱에 있고, 승객은 없는 구조.

7. **Addressable의 정의** — "디스크의 에셋을 주소로 참조해서 필요할 때만 메모리에 로드하는 수동 제어 시스템". 일반 참조(자동)와 대비되는 개념.
