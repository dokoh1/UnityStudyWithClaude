**"이 문제에 이 패턴이 필요한가?"를 먼저 자문. 패턴을 위한 패턴은 오버엔지니어링.**

## 1. Singleton — 전역 매니저

### 언제 쓰는가
- 게임 전체에서 딱 1개, 어디서든 접근 필요
- GameManager, AudioManager, InputManager, SaveManager
- **2~3개로 제한**, 남용하면 의존성 지옥

### 언제 쓰면 안 되는가
- "편해서" 전부 싱글톤 → 스파게티
- 데이터 공유만 필요 → ScriptableObject 변수 패턴
- 테스트 필요한 로직 → DI로 대체

### 시니어 관점: 싱글톤의 진짜 문제

**1. 암묵적 의존성** — 코드 깊숙이 `GameManager.Instance` 호출이 숨어있음. 클래스 선언만 봐서는 의존성 파악 불가.

**2. 순환 의존성 감지 불가** — A.Instance가 B.Instance를 호출, B.Instance가 A.Instance를 호출해도 컴파일 에러 안 남. 런타임에만 문제 발생.

**3. 싱글톤이 싱글톤을 낳음** — 6개월 후 15개 싱글톤이 서로 참조하는 상황.

### 리드의 대안

```csharp
// 생성자 주입 (DI) — 의존성이 명시적
public class QuestSystem
{
    private readonly IRewardService rewardService;
    private readonly INotificationService notificationService;

    public QuestSystem(IRewardService reward, INotificationService notification)
    {
        this.rewardService = reward;
        this.notificationService = notification;
    }
}
// → 클래스 선언만 봐도 의존성 파악
// → 순환 의존성이면 컴파일 에러
// → Mock 객체로 테스트 가능
```

### 규모별 판단
- 소규모: 싱글톤 OK (빠른 개발)
- 중규모: 핵심 2~3개만, 나머지 SO 참조
- 대규모: DI 컨테이너(VContainer)로 대체

## 2. Observer — 이벤트 시스템

### 언제 쓰는가
- A가 일어나면 B, C, D가 반응, A는 B/C/D를 모름
- 플레이어 사망 → UI, 오디오, 통계 등 반응
- 적 처치 → 점수, 퀘스트, 이펙트
- 1:N 관계, 느슨한 결합

### 3가지 방법의 선택
- **C# event/Action** — 같은 프리팹 내부, 가장 빠름
- **UnityEvent** — Inspector 드래그&드롭, UI 버튼
- **SO 이벤트** — 씬을 넘나드는 전역, 매니저 간 통신

### 시니어 관점: 대규모에서의 이벤트 관리

SO 이벤트 50개+ → 추적 어려움. 대안: **struct 기반 EventBus**

```csharp
// 타입 안전, GC 없음, 코드로 추적 가능
public struct DamageEvent { public GameObject target; public int amount; }

EventBus.Subscribe<DamageEvent>(OnDamage);
EventBus.Publish(new DamageEvent { target = enemy, amount = 50 });
```

### 이벤트 아키텍처 선택
- 소규모: C# event + SO 이벤트
- 중규모: SO 이벤트 주력
- 대규모: struct EventBus 주력 + C# event (내부)

## 3. State — 상태 머신

### 언제 쓰는가
- 오브젝트가 여러 상태, 상태마다 행동 다름, 전환 규칙 복잡
- 적 AI (Idle, Patrol, Chase, Attack, Flee, Dead)
- 플레이어 (Idle, Run, Jump, Attack, Climb)
- UI 화면 전환 (MainMenu, InGame, Pause, GameOver)
- 보스 페이즈, 게임 진행 상태

### 언제 쓰면 안 되는가
- 상태 2개뿐 → bool
- 전환 단순 → enum + switch
- 상태가 동시에 공존 → 계층적 상태 머신(HFSM)

### 시니어 관점: 계층적 상태 머신 (HFSM)

상태가 20개+ → 단순 FSM 한계. 상태를 그룹화:

```
[Combat] (상위)
  ├── Chase
  ├── Attack (하위: Melee, Ranged, Special)
  └── Flee
[NonCombat] (상위)
  ├── Idle
  └── Patrol
```

### Behavior Tree (AAA급)

State Machine보다 유연:
- 트리 구조로 행동 조합
- 조건 → 행동 순서를 직관적 표현
- 우선순위 기반 의사결정

### 선택 기준
- 상태 5~10개 → State Machine
- 상태 그룹화 필요 → HFSM
- 복잡한 조건부 행동 → Behavior Tree
- AI가 목표 세우고 계획 → GOAP

## 4. Command — 행동을 객체로

### 언제 쓰는가
- 행동을 기록하고 되돌리거나 재실행
- 레벨 에디터 Undo/Redo
- 턴제 게임 이동 되돌리기
- 리플레이 시스템
- 입력 리매핑, 매크로

### 언제 쓰면 안 되는가
- Undo 필요 없는 단순 호출
- 실시간 액션 (오버헤드)

### 시니어 관점: 네트워크 + 리플레이

```csharp
// 커맨드를 직렬화해서 네트워크 전송 → 롤백 넷코드의 기초
[Serializable]
public struct MoveCommand : ICommand
{
    public int playerId;
    public Vector3 direction;
    public int frameNumber;

    public void Execute(GameState state) { ... }
    public void Undo(GameState state) { ... }
}

// 로컬 예측 → 서버 확인 → 불일치 시 Undo → 서버 상태로 덮어쓰기
```

리플레이: 입력을 커맨드로 기록 → 타임스탬프와 함께 저장 → 재생. 레이싱 고스트, 하이라이트.

커맨드 + Object Pool: 커맨드 객체 자체를 풀링해서 GC 방지.

## 5. Strategy — 알고리즘 교체

### 언제 쓰는가
- 같은 동작을 여러 방식으로, 런타임 교체 가능
- 데미지 계산 (일반, 크리티컬, 속성, 고정)
- 이동 방식 (걷기, 날기, 수영)
- AI 행동 (공격적, 방어적, 도주)
- 저장 방식 (JSON, Binary, Cloud)

### 시니어 관점: SO 기반 Strategy

```csharp
public abstract class DamageStrategy : ScriptableObject
{
    public abstract int Calculate(AttackData atk, DefenseData def);
}

[CreateAssetMenu] public class PhysicalDamage : DamageStrategy { ... }
[CreateAssetMenu] public class MagicalDamage : DamageStrategy { ... }
[CreateAssetMenu] public class TrueDamage : DamageStrategy { ... }
```

Inspector에서 스킬마다 다른 Strategy .asset 연결 → 코드 변경 없이 새 데미지 타입 추가. 밸런스 조정 = Inspector에서 값 변경.

### Strategy 체이닝

```csharp
public class SkillData : ScriptableObject
{
    public DamageStrategy damageStrategy;
    public EffectStrategy[] additionalEffects;  // Burn, Slow, HealOnHit...
}
// 화염 참격 = PhysicalDamage + BurnEffect + SlowEffect
// 코드 변경 없이 조합!
```

## 6. Factory — 생성 로직 통합

### 언제 쓰는가
- 타입에 따라 다른 오브젝트 생성
- 생성 시 초기화 로직 있음
- 여러 곳에서 중복 생성 코드

### 예시
- 적 스폰 (타입별 프리팹), 아이템 생성 (ID → 아이템), 스킬 이펙트, UI 팝업

### Object Pool과 결합
Factory가 Pool에서 꺼내서 초기화 후 반환.

---

## 7. Object Pool — 반복 생성/파괴 최적화

### 언제 쓰는가
- 짧은 시간에 반복 생성/파괴: 총알, 이펙트, 적 웨이브, 데미지 숫자

### 언제 안 쓰는가
- 한 번만 생성 (보스), 고정 환경, 수 5개 이하

---

## 8. MVP — UI와 로직 분리

### 언제 쓰는가
- UI와 게임 로직 분리 필요
- 같은 데이터를 여러 UI에서 표시
- 인벤토리, 상점, 스탯 화면, 설정

```
Model: 순수 데이터
View: UI 표시만 (로직 없음)
Presenter: Model과 View 연결
```

## 전체 패턴 선택 가이드

| 상황 | 패턴 |
|---|---|
| 전역 매니저 | Singleton (최소) 또는 DI |
| A 발생 시 B,C,D 반응 | Observer |
| 여러 상태 + 전환 | State |
| Undo/Redo/기록 | Command |
| 같은 동작, 다른 방식 | Strategy |
| 생성 로직 통합 | Factory |
| 반복 생성/파괴 | Object Pool |
| UI와 로직 분리 | MVP |

## 규모별 권장 패턴

### 게임잼 / 프로토타입
Singleton + Observer + Object Pool

### 소규모 인디
Singleton(2~3개) + C# event + State + Object Pool + Factory

### 중규모
Singleton 최소→SO 참조 + SO 이벤트 + State/Strategy + Command(필요 시) + Factory + Pool + MVP

### 대규모
DI(VContainer) + EventBus(struct) + State/BT + Command + Strategy(SO) + Factory + Pool + MVP/MVVM

## 리드 레벨: 아키텍처 설계

### 실제 중규모 RPG 아키텍처

```
Bootstrap → DI Container → Services → Events → Presentation

서비스 레이어 (순수 C#):
  GameService, CombatService(Strategy), QuestService(Observer),
  InventoryService(Command), SaveService(Strategy)

이벤트: EventBus(struct) + C# event(내부)
데이터: ScriptableObject + Addressable
UI: MVP 패턴
AI: State Machine 또는 Behavior Tree
```

### 리드의 도입 판단 체크리스트

```
□ 이 패턴 없이 코드가 복잡해지는가?
□ 팀원이 이해할 수 있는가?
□ 유지보수가 쉬워지는가?
□ 테스트가 가능해지는가?
□ 과도한 추상화가 아닌가?
→ 하나라도 "아니오"면 도입 안 함
```

### 팀 규모별
- 1~2명: 빠른 개발 우선
- 3~5명: 팀 내 규칙 필요
- 5~10명: 아키텍처 문서 필요
- 10명+: 아키텍처 팀/리드 + 자동 테스트 + CI/CD

## 오답 노트

1. **싱글톤의 진짜 문제는 "암묵적 의존성"** — 코드를 전부 읽어야 의존성 파악. 생성자 주입(DI)이면 선언만 봐도 파악. 5개+ 싱글톤은 아키텍처 재검토 시그널.

2. **SO 이벤트는 대규모에서 추적 어려움** — 50개+ SO 이벤트 파일 관리 어려움. struct EventBus가 타입 안전하고 IDE에서 검색 가능.

3. **State Machine의 한계** — 상태 20개+ → HFSM(계층적). 복잡한 조건부 → Behavior Tree. AI가 목표 설정 → GOAP.

4. **Command는 Undo만이 아님** — 네트워크 동기화(롤백 넷코드), 리플레이, 입력 리매핑, 매크로에도 활용.

5. **Strategy + SO = 데이터 주도 설계** — Inspector에서 스킬 조합, 코드 변경 없이 새 타입 추가. Q5의 SkillEffect가 이 패턴.

6. **패턴은 도구, 목적이 아님** — "이 패턴 없이 코드가 복잡해지는가?"가 도입 기준. 프로젝트 규모와 팀 역량에 맞춤.

7. **리드의 핵심 원칙: 의존성은 단방향** — Bootstrap → Services → Events → Presentation. 상위가 하위를 알되 하위는 상위를 모르게.
