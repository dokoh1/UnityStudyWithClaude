## Chapter 1: Coroutine — Unity의 가짜 비동기

### 원리

코루틴은 멀티스레드가 아니다. 메인 스레드에서 "잠깐 멈췄다가 다음 프레임에 이어서" 실행하는 것이다.

```csharp
IEnumerator ShootSequence()
{
    SpawnBullet();
    yield return new WaitForSeconds(0.5f);  // "0.5초 후에 이어서"
    PlaySound();
}
```

yield return을 만나면 함수 실행이 멈추고, Unity 엔진에 "이 조건이 되면 깨워줘"라고 등록한다. 게임은 계속 돌아간다.

### 내부 동작: "2초 대기"는 어떻게 동작하는가?

`yield return new WaitForSeconds(2f)`가 실행되면 내부적으로 `targetTime = Time.time + 2f`가 저장된다. 현재 시간이 10.5초면 targetTime은 12.5초. 그 뒤 Unity의 코루틴 스케줄러가 매 프레임 `Time.time >= targetTime`을 체크한다. 12.5초가 되면 코루틴을 이어서 실행한다.

`yield return null`은 시간을 체크하지 않는다. 코루틴 스케줄러의 UpdateCoroutines()가 매 프레임 호출되므로, null이면 "다음 호출 = 다음 프레임"에 무조건 이어서 실행한다.

### 핵심: 코루틴은 "매 프레임 처음부터 다시 실행"하지 않는다

yield return에 도달하면 함수 실행이 **멈춘다.** 코루틴은 "마지막으로 yield한 위치"를 기억하고, 조건이 충족되면 그 위치에서 이어서 실행한다. 2초 대기 중에 yield return 줄에 다시 "도달"하지 않으므로 WaitForSeconds가 추가 생성되지도 않는다.

단, while 루프 안에서 yield return을 쓰면 매번 다시 도달하므로 캐싱이 필요하다.

### Coroutine의 문제점

1. **GC 할당** — `yield return new WaitForSeconds()` 매번 힙 할당. 캐싱으로 해결 가능.
2. **반환값 없음** — IEnumerator는 값을 return할 수 없다.
3. **에러 처리 어려움** — try-catch가 yield를 감쌀 수 없다.
4. **MonoBehaviour 종속** — 순수 C# 클래스에서 사용 불가. 오브젝트 파괴 시 코루틴도 중단.

## Chapter 2: async/await — C# 표준 비동기

### Coroutine의 문제를 해결

```csharp
async Task<string> LoadDataAsync()
{
    try
    {
        var result = await webRequest.SendWebRequest();
        return result.downloadHandler.text;  // 반환값 가능!
    }
    catch (Exception e)
    {
        Debug.LogError(e);  // 에러 처리 가능!
        return null;
    }
}
```

### Unity에서의 문제

여기서 첫 번째 큰 의문이 생겼다.

> "Task는 스레드풀을 쓰는데 그러면 병렬 아니야? 메인 스레드에서 실행하지 않는 게 왜 문제야?"

이 질문을 이해하려면 먼저 **워커 스레드**가 뭔지 알아야 했다.

## 사이드퀘스트: 워커 스레드란?

프로그램이 실행되면 "프로세스"가 만들어지고, 프로세스 안에서 코드를 실행하는 단위가 "스레드"다. Unity 프로세스 안에는 여러 스레드가 있다.

- **메인 스레드** = "사장님". 모든 Unity API를 다룰 수 있는 유일한 스레드.
- **워커 스레드** = "알바". 사장님이 시킨 계산만 하고, Unity API(Transform, GameObject 등)는 만지지 못한다.

워커가 Unity API를 못 쓰는 이유: 사장님(메인)이 기계를 조작 중인데 알바(워커)가 동시에 같은 기계를 만지면 기계가 고장난다 (레이스 컨디션). 여러 스레드가 동시에 같은 Transform을 수정하면 최종 값이 뭔지 모르고, 파괴된 오브젝트에 접근하면 크래시가 난다. 모든 접근에 Lock을 걸면 성능이 떨어지니까, Unity는 "메인 스레드에서만 쓰라"고 강제한다.

**스레드풀**은 워커 스레드를 미리 만들어놓고 재사용하는 "알바 대기실"이다. Task.Run()을 호출하면 스레드풀의 빈 워커에 작업을 배정하고, 끝나면 워커가 대기 상태로 복귀한다.

### Task는 병렬인가?

Task.Run()은 진짜 다른 스레드에서 실행되므로 병렬이다. 하지만 `await Task.Delay()`나 `await httpClient.GetStringAsync()` 같은 I/O 작업은 워커 스레드를 쓰지 않는다.

### "OS가 처리한다"는 게 뭔 뜻이야?

HTTP 요청을 보내면 OS 커널이 네트워크 카드(하드웨어)에 전달하고, CPU는 아무것도 안 한다. 하드웨어가 응답을 받으면 OS가 앱에 "왔다!"고 알려준다. 택배를 보내고 현관 앞에서 기다리는 대신 다른 일하다가 "택배 도착" 알림을 받는 것과 같다. 프로세스를 새로 만드는 게 아니라, OS에 부탁하고 알림을 받는 구조다.

### 메인 스레드가 아닌 게 왜 문제인가

```csharp
async void Start()
{
    await Task.Run(() => { ... });  // 워커 스레드에서 실행
    transform.position = Vector3.zero;  // ⚠️ 어느 스레드에서 실행?
}
```

await 이후 코드가 메인 스레드가 아닌 워커 스레드에서 실행될 수 있다. Unity API는 메인 스레드에서만 호출 가능하므로 크래시가 난다.

## 사이드퀘스트: SynchronizationContext를 왜 잃어버리는가?

SynchronizationContext는 "await 끝나면 어디로 돌아갈지" 정보다. 출장 갈 때 "돌아올 사무실 주소"를 적어놓는 것과 같다.

메인 스레드에서 async 메서드를 시작하면 UnitySynchronizationContext가 캡처된다. "await 끝나면 메인 스레드로 보내줄게"라는 약속이다.

하지만 `Task.Run(async () => { ... })` 안에 들어가면 **워커 스레드에는 Context가 없다.** 그 안에서 `await Task.Delay()`를 하면 "돌아갈 곳"을 찾는데 Context가 null이라 스레드풀의 아무 워커에서 이어서 실행된다. 이 "잃어버린 상태"가 Task.Run 밖으로 전파될 수 있다.

UniTask는 SynchronizationContext에 의존하지 않고 Unity의 PlayerLoop에 직접 콜백을 등록한다. "다음 프레임에 메인 스레드에서 실행"을 100% 보장한다.

## Chapter 3: UniTask — 두 문제를 모두 해결

### UniTask가 해결하는 것

| Task의 문제 | UniTask의 해결 |
|---|---|
| 스레드풀 사용 → Unity API 접근 불가 | 메인 스레드에서 실행 (기본) |
| Task는 class → 힙 할당 → GC | UniTask는 struct → 스택 → GC 0 |
| 취소가 어려움 | CancellationToken 통합, GetCancellationTokenOnDestroy() |
| Unity 시간/프레임 개념 없음 | UniTask.Delay, Yield, WaitForFixedUpdate 등 전부 지원 |

```csharp
async UniTask LoadAndDisplay(CancellationToken ct)
{
    await UniTask.Delay(2000, cancellationToken: ct);  // Unity 시간, 메인 스레드
    try
    {
        var www = UnityWebRequest.Get(url);
        await www.SendWebRequest().WithCancellation(ct);
        uiText.text = www.downloadHandler.text;  // 메인 스레드, 안전!
    }
    catch (OperationCanceledException) { }  // 씬 전환 시 자동 취소
    catch (Exception e) { Debug.LogError(e); }
}
```

UniTask.RunOnThreadPool을 쓰면 무거운 계산을 워커에서 하고, await 이후 자동으로 메인 스레드로 복귀한다. Task.Run과 달리 PlayerLoop에 직접 콜백을 등록하므로 100% 보장된다.

### 언제 뭘 쓰는가

- **Coroutine** — 간단한 타이밍 제어, 프로토타이핑
- **UniTask** — 서버 통신, 에셋 로드, 반환값/에러 처리가 필요한 프로덕션 코드
- **Task** — Unity에서는 가급적 사용하지 않음. 순수 C# 라이브러리 연동 시만.

## Chapter 4: 멀티스레드의 세계

여기까지는 전부 메인 스레드 1개에서 일어나는 일이었다. 이제 진짜로 여러 스레드에서 동시에 실행하는 이야기.

### C# Thread — 직접 스레드 생성

```csharp
Thread thread = new Thread(() =>
{
    // 워커 스레드 — Unity API 불가, 순수 계산만
    var path = CalculatePath(start, end);
    results.Enqueue(path);  // ConcurrentQueue로 결과 전달
});
thread.Start();
```

문제: 스레드 생성 비용 큼, 동기화 수동 관리, 레이스 컨디션/데드락 위험.

## 사이드퀘스트: Lock, Mutex, ConcurrentQueue

멀티스레드에서 같은 데이터를 동시에 수정하면 레이스 컨디션이 발생한다. `score += 10`은 실제로 "읽기 → 더하기 → 쓰기" 3단계인데, 두 스레드가 동시에 "읽기"를 하면 한쪽의 결과가 덮어씌워진다.

**Lock** — 1인용 화장실 잠금장치. `lock(obj) { ... }` 블록 안에는 한 번에 하나의 스레드만 들어갈 수 있다. 다른 스레드는 잠금이 풀릴 때까지 대기한다. 문제: 성능 저하(대기 중 스레드가 놀음), 데드락(두 스레드가 서로의 Lock을 기다리면 영원히 멈춤).

**Mutex** — Lock의 OS 버전. 다른 프로세스 간에도 동기화 가능하지만, Unity에서는 lock으로 충분하다.

**ConcurrentQueue** — Lock 없이 안전한 큐. 여러 스레드가 동시에 Enqueue/Dequeue해도 안전하다. 워커 스레드에서 계산 결과를 넣고, 메인 스레드 Update()에서 꺼내서 Unity에 적용하는 패턴에 쓴다.

---

### Task.Run — 스레드풀 활용

Thread보다 나은 점: 스레드를 직접 만들지 않고 스레드풀에서 재사용. async/await와 자연스럽게 연결. 하지만 여전히 Unity API 사용 불가, 메인 스레드 복귀 보장 안 됨.

## Chapter 5: Unity Job System

### 기존 멀티스레드의 문제를 전부 해결

Job System은 Unity가 워커 스레드를 자동 관리하고, NativeArray로 GC 없는 안전한 데이터 공유를 하며, 컴파일 타임에 레이스 컨디션을 감지한다.

```csharp
[BurstCompile]
struct MoveJob : IJobParallelFor
{
    public NativeArray<Vector3> positions;
    [ReadOnly] public NativeArray<Vector3> velocities;
    public float deltaTime;

    public void Execute(int index)
    {
        positions[index] += velocities[index] * deltaTime;
    }
}

// 10000개를 64개씩 나눠서 워커 스레드에 분배
JobHandle handle = job.Schedule(10000, 64);
handle.Complete();
```

## 사이드퀘스트: Burst, NativeArray, Mono/IL2CPP/LLVM

> "BurstCompile가 뭐야? C++급 성능이란 건 뭐야? NativeArray는? Mono/IL2CPP/LLVM은?"

### C# 코드가 실행되기까지

C# 소스코드는 먼저 IL(중간 언어)로 컴파일된다. 이 IL을 기계어로 바꾸는 방식이 여러 가지다.

**Mono** — 실행 중에 IL을 기계어로 변환 (JIT, Just-In-Time). 빌드가 빠르지만 실행이 느리고, iOS에서는 JIT가 금지되어 사용 불가. 에디터/개발 중에 사용.

**IL2CPP** — 빌드 시점에 IL을 C++ 코드로 변환하고, C++ 컴파일러가 기계어로 만듦 (AOT, Ahead-Of-Time). 빌드는 느리지만 실행이 빠르고 iOS에서도 동작. 빌드/배포 시 사용.

**LLVM** — 컴파일러를 만들기 위한 프레임워크. C, C++, Rust, Swift 등 여러 언어가 LLVM을 통해 최적화된 기계어를 생성한다. SIMD 자동 벡터화, 루프 최적화, 인라이닝 등 강력한 최적화를 수행한다.

### Burst Compiler

Burst는 Job System의 C# 코드를 LLVM IR로 변환하고, LLVM이 최적화해서 네이티브 기계어를 생성한다.

핵심 최적화: **SIMD(Single Instruction, Multiple Data)**. 일반 연산은 float 1개씩 처리하지만, SIMD는 한 번에 4~8개를 동시에 처리한다. 같은 C# 코드가 10~50배 빠르게 실행된다.

### "C++급 성능"이란?

일반 C#(Mono)을 1x로 보면, IL2CPP는 ~1.5~2x, Burst는 ~10~50x, C++ 네이티브도 ~10~50x. Burst와 C++이 비슷한 이유는 둘 다 LLVM을 사용하기 때문이다.

### NativeArray

일반 C# 배열은 힙에 할당되어 GC가 관리한다. NativeArray는 비관리 메모리에 할당되어 GC의 대상이 아니다. 메모리 주소가 고정되고, 연속된 메모리 배치로 캐시 친화적이며, Burst와 Job System이 안전하게 접근할 수 있다. 대신 개발자가 Dispose()를 직접 호출해서 해제해야 한다.

### LLVM이 빠르면 왜 전체 빌드에서 안 쓰는가?

Burst(LLVM)는 C#의 아주 작은 부분집합만 컴파일할 수 있다. class, string, LINQ, delegate, virtual, try-catch, GC 관리 객체, Unity API 전부 불가. struct, NativeArray, 기본 수학 연산만 가능하다. C#의 대부분의 기능이 GC에 의존하는데 LLVM은 GC를 모르기 때문이다. 또한 컴파일 시간이 길어서 개발 중에는 Mono를 쓴다.

### Burst와 Job System의 관계

둘은 독립적이지만 같이 쓸 때 시너지가 극대화된다.

- **Job System만** — 4코어 병렬 = ~4배. 하지만 각 코어의 코드가 느린 C#.
- **Burst만** — 1코어에서 ~10배. 하지만 코어 1개만 사용.
- **둘 다** — 4코어 × 코어당 10배 = ~40배! "빠른 코드를 4명이 나눠서 하는 것."

Burst는 Job System의 IJob 구조체가 "class 없음, GC 없음, NativeArray만" 조건을 만족하기 때문에 컴파일 가능하다.


## 오답 노트

이 세션에서 틀린 문제는 없었으나, 새로 정리된 인사이트:

1. **비동기 ≠ 멀티스레드** — Coroutine, async/await, UniTask는 전부 메인 스레드 1개에서 돌아간다. "기다리는 동안 다른 일 가능"이지 "동시에 실행"이 아니다.

2. **워커 스레드는 메인 스레드가 아니다** — Unity API는 메인 스레드에서만 사용 가능. 워커에서 Transform을 만지면 크래시. 이건 Unity가 레이스 컨디션을 방지하기 위해 의도적으로 막은 것.

3. **SynchronizationContext를 잃어버리는 이유** — Task.Run 안은 워커 스레드이고 워커에는 Context가 없다. 그 안에서 await하면 "돌아갈 곳"이 없어서 스레드풀에 남는다. UniTask는 PlayerLoop에 직접 등록하므로 이 문제가 없다.

4. **"OS가 처리한다" = 별도 프로세스가 아니다** — OS 커널이 하드웨어(네트워크 카드 등)에 전달하고 완료 알림을 받는 것. CPU 스레드가 기다리는 게 아니라 하드웨어가 처리.

5. **Burst가 전체 C#을 대체하지 못하는 이유** — class, string, GC 등 C#의 핵심 기능을 지원하지 못한다. LLVM은 GC를 모르기 때문. Job System의 struct + NativeArray 환경에서만 동작한다.

6. **Job System과 Burst의 관계** — Job System = 멀티코어 병렬 (~4배), Burst = 단일 코어 최적화 (~10배). 둘을 합치면 ~40배. 독립적이지만 같이 쓸 때 시너지가 극대화된다.
